# Java.lang() package
java.lang 패키지는 자바 프로그래밍의 **가장 기본이 되는 클래스들**을 담고 있는 핵심 패키지입니다. 너무나 필수적이라서 모든 자바 파일에 import java.lang.*; 문이 자동으로 포함되어 있습니다. 즉, 별도의 임포트 없이 바로 사용할 수 있죠.
주요 구성 요소와 특징을 알기 쉽게 정리해 드릴게요.

### 1\. 주요 클래스 구성
이 패키지에는 우리가 자바를 쓰면서 숨 쉬듯 사용하는 클래스들이 모여 있습니다.
여기 앞부분에 설명이 있다면 반드시 한 줄을 비워주세요.

| 분류 | 주요 클래스 | 설명 |
|:---:|:---|:---|
| **최상위 클래스** | Object | 모든 자바 클래스의 조상. equals(), hashCode(), toString() 등을 제공. |
| **문자열 처리** | String, StringBuffer, StringBuilder | 텍스트 데이터를 다루는 클래스들. |
| **기본형의 객체화** | Byte, Short, Integer, Long, Float, Double, Character, Boolean | **Wrapper 클래스**라고 하며, 기본 자료형을 객체로 다룰 때 사용. |
| **수학/시스템** | Math, System, Runtime | 수학 계산, 시스템 속성 읽기, 가비지 컬렉션 호출 등. |
| **스레드/프로세스** | Thread, Runnable, Process | 멀티스레딩과 외부 프로세스 실행 제어. |
| **예외 처리** | Throwable, Error, Exception | 프로그램 실행 중 발생하는 오류와 예외의 최상위 클래스. |

### 2\. 핵심 클래스 깊게 보기
**Object: 자바의 뿌리**
자바의 모든 클래스는 명시하지 않아도 Object를 상속받습니다. 덕분에 어떤 객체든 공통된 메서드를 사용할 수 있습니다.
* equals(Object obj) : 두 객체가 같은지 비교 (주로 재정의해서 사용)
* toString() : 객체의 문자열 정보를 반환

⠀**String vs StringBuilder**
* **String**: 불변(Immutable)입니다. 한 번 만든 문자열은 변하지 않으며, 내용을 수정하면 새로운 객체가 생성됩니다.
* **StringBuilder**: 가변(Mutable)입니다. 문자열을 자주 변경(추가, 삭제)해야 할 때 메모리 효율이 훨씬 좋습니다.

⠀**System 클래스**
표준 입출력이나 시스템의 현재 시간을 가져올 때 사용합니다.
* System.out.println() : 콘솔 출력
* System.currentTimeMillis() : 현재 시간을 밀리초 단위로 반환

⠀
### 3\. 상속 구조 (Hierarchy)
java.lang 내의 예외 처리는 자바 프로그램의 안정성을 책임지는 중요한 계층 구조를 가집니다.
**Tip:** 자바 9 버전부터는 모듈 시스템이 도입되면서 java.lang은 java.base 모듈에 포함되어 관리됩니다. 여전히 임포트는 필요 없지만, 내부 구조적으로는 더 견고해졌죠.

자바에서 Object **다형성**이란, 모든 클래스의 최상위 부모인 Object 타입을 이용해 **어떤 타입의 객체든 참조할 수 있는 능력**을 말합니다.
자바의 모든 객체는 결국 Object이기도 하다는 점을 이용한 프로그래밍 기법이죠.

### 1\. Object 다형성의 핵심 원리
자바에서는 부모 타입의 참조 변수가 자식 타입의 인스턴스를 참조할 수 있습니다. Object는 모든 클래스의 부모이므로, 다음과 같은 코드가 가능합니다.

``` java
Object obj1 = new Dog();    // Dog 인스턴스를 Object 타입으로 참조
Object obj2 = new Car();// Car 인스턴스를 Object 타입으로 참조
Object obj3 = "Hello";  // String 인스턴스를 Object 타입으로 참조
```

### 2\. 왜 Object 다형성을 사용할까요?
가장 큰 이유는 **범용성(Generality)** 때문입니다. 어떤 타입이 들어올지 모르거나, 여러 타입을 동시에 관리해야 할 때 유용합니다.
* **다양한 객체 배열 관리**: 서로 다른 타입의 객체들을 하나의 배열에 담을 수 있습니다.Java### Object[] objects = { new Dog(), new Car(), "String Data" };
* 
* **매개변수의 유연성**: 메서드 작성 시 어떤 객체든 인자로 받고 싶을 때 사용합니다. (예: System.out.println(Object x))

⠀
### 3\. Object 다형성의 한계와 다운캐스팅 (Downcasting)
### Object 타입으로 객체를 참조하면 편리하지만, 큰 **제약 사항**이 하나 있습니다. Object **클래스에 정의된 메서드(**toString**,** equals **등)만 바로 사용할 수 있다**는 점입니다.
* **문제**: Object obj = new Dog();라고 선언하면, obj.bark() 같은 Dog만의 메서드는 호출할 수 없습니다. 컴파일러는 obj를 그저 '객체'로만 보기 때문입니다.
* **해결**: 실제 객체의 기능을 사용하려면 다시 원래 타입으로 변환하는 **다운캐스팅**이 필요합니다.

``` java
if (obj instanceof Dog) {   // 1. 타입 확인 (안전장치)
	Dog myDog = (Dog) obj;  // 2. 다운캐스팅
     myDog.bark();           // 3. Dog 전용 메서드 사용 가능
}

```

### 4\. 현대 자바에서의 변화: 제네릭(Generics)
과거에는 Object 다형성을 이용해 데이터를 주고받았지만, 매번 형변환(Casting)을 해야 하고 런타임 오류 위험이 컸습니다. 그래서 **자바 5**부터는 **제네릭(Generics)**이 등장하여 Object 다형성의 단점을 보완하게 되었습니다.
**요약**: Object 다형성은 "무엇이든 담을 수 있는 만능 바구니"와 같지만, 꺼내서 쓸 때는 내용물이 무엇인지 정확히 확인하고 이름표(캐스팅)를 붙여줘야 합니다.

java.lang.Object 클래스가 가진 메서드 중 가장 친숙한 것이 바로 toString()일 겁니다. 이 메서드의 본질은 **"객체의 상태를 문자열로 표현하는 것"**입니다.
하지만 기본 동작 방식과 우리가 실제로 기대하는 모습에는 차이가 좀 있습니다.

### 1\.toString()의 기본 동작 (Default)
Object 클래스에 정의된 원래의 toString()은 다음과 같은 형식으로 문자열을 리턴합니다.
**클래스이름 @ 16진수_해시코드**
예를 들어, MyClass라는 객체를 생성해 출력하면 MyClass@1a2b3c4d 같은 정체를 알 수 없는 문자열이 나옵니다. 이는 개발자에게 객체의 **내용(데이터)**을 알려주기에는 부족합니다.

### 2\. 왜 오버라이딩(재정의)을 할까?
우리가 궁금한 건 "이 객체의 메모리 주소"가 아니라, **"이 객체가 가진 데이터가 무엇인가"**이기 때문입니다.
디버깅을 하거나 로그를 남길 때, toString()을 재정의해두면 객체 이름만 출력해도 내부 정보를 바로 확인할 수 있습니다.

``` java
public class Student {
    private String name = "홍길동";
    private int age = 20;

    // toString을 재정의하여 의미 있는 정보 반환
    @Override
    public String toString() {
        return "Student[name=" + name + ", age=" + age + "]";
    }
}

// 실행 코드
Student s = new Student();
System.out.println(s); // 출력: Student[name=홍길동, age=20]

```

### 3\.toString()의 특징과 팁
* **자동 호출**: System.out.println(obj)처럼 객체를 출력문에 넣거나, 문자열 결합("Hello " + obj)을 할 때 자바가 내부적으로 obj.toString()을 자동으로 호출합니다.
* **IDE 활용**: IntelliJ나 Eclipse 같은 도구에서 Alt + Insert (또는 Cmd + N) 단축키를 누르면 모든 필드를 포함한 toString()을 자동으로 생성해 줍니다.
* **디버깅의 핵심**: 복잡한 시스템에서 에러가 났을 때, 객체의 상태를 한눈에 볼 수 있게 toString()이 잘 구현되어 있으면 원인 파악이 훨씬 빠릅니다.

⠀
### 4\. 실전 팁:Arrays.toString()
배열(Array)은 일반 객체와 달라서 직접 toString()을 호출하면 원하는 값이 나오지 않습니다. 이때는 java.util.Arrays 클래스의 도움을 받아야 합니다.

``` Java
int[] nums = {1, 2, 3};
System.out.println(nums);             // 출력: [I@7ad041f3 (알 수 없는 값)
System.out.println(Arrays.toString(nums)); // 출력: [1, 2, 3] (정확한 값)
```

Object 클래스의 equals() 메서드는 **"두 객체가 같은가?"**를 판단하는 아주 중요한 도구입니다. 하지만 자바에서 '같다'는 의미는 두 가지로 나뉘기 때문에 주의가 필요합니다.

### 1\. 동일성(Identity) vs 동등성(Equality)
equals()를 제대로 이해하려면 이 두 개념을 먼저 구분해야 합니다.
* **동일성 (****==** **연산자)**: 두 객체가 **메모리상의 같은 주소**를 가리키고 있는지 확인합니다. (완전히 같은 녀석인가?)
* **동등성 (****equals()** **메서드)**: 두 객체의 **내용(값)**이 같은지 확인합니다. (서로 다른 녀석이라도 데이터가 같은가?)

⠀
### 2\.Object.equals()의 기본 동작
아이러니하게도, Object 클래스에 정의된 **기본** **equals()****는 내부적으로** **==** **연산자를 사용**합니다.

``` java
// Object 클래스의 실제 내부 코드 (대략적)
public boolean equals(Object obj) {
    return (this == obj);
}
```

즉, 여러분이 직접 만든 클래스에서 equals()를 **오버라이딩(재정의)**하지 않으면, equals()는 ==와 똑같이 주소값만 비교하게 됩니다.

### 3\. 왜 오버라이딩이 필요한가?
우리가 만든 Student 클래스에서 학번이 같으면 '같은 학생'이라고 간주하고 싶을 때가 있습니다. 이때 equals()를 재정의합니다.

``` java
public class Student {
    private int id;
    private String name;

    public Student(int id, String name) {
        this.id = id;
        this.name = name;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true; // 주소가 같으면 당연히 같음
        if (o == null || getClass() != o.getClass()) return false; // null이거나 클래스 타입이 다르면 다름
        Student student = (Student) o;
        return id == student.id; // 학번이 같으면 같은 학생으로 인정!
    }
}
```


### 4\.equals()를 재정의할 때의 철칙
자바 규약에 따라 equals()를 재정의할 때는 반드시 다음 규칙을 지켜야 안전합니다.
**1** **반사성**: x.equals(x)는 항상 true여야 함.
**2** **대칭성**: x.equals(y)가 true면 y.equals(x)도 true여야 함.
**3** **추이성**: x=y, y=z가 true면 x=z도 true여야 함.
**4** **일관성**: 객체가 변하지 않았다면 결과는 항상 같아야 함.
**5** **hashCode()****와 세트**: **매우 중요!** equals()를 재정의했다면 hashCode()도 반드시 함께 재정의해야 합니다. 그렇지 않으면 HashMap이나 HashSet 같은 컬렉션에서 객체를 찾지 못하는 대참사가 발생합니다.

⠀
### 5\. 실전 팁: String은 이미 되어 있다!
우리가 문자열을 비교할 때 str1 == str2가 아니라 str1.equals(str2)를 쓰는 이유가 바로 이것입니다. String 클래스는 이미 **내용을 비교하도록** **equals()****를 오버라이딩**해두었기 때문입니다.
